Api gataway provides several important functions:

- Authentication
- Security policy enforcement
- Load balancing and circuit breaking (Circuit breaking is a pattern that stops sending requests to a service that is failing, so it doesn't become even more overloaded.)
- Protocol Translation and service discovery
- Monitoring, Logging, analytics and billing
- Caching

Step 1: Client send a request to api gataway. request is typically HTTP based. It can be REST, GRAPHQL or some other high level abstraction.

Step 2: Api Gataway validates the HTTP request.

Step 3: Api Gataway checks callers ip address and other HTTP Headers against its allow list and deny list.

Step 4: Api Gataway passes request to identity provider for authentication and authorization. Api Gataway receives an authenticated session back from the provider with the scope of what the request is allowed to do.

Step 5: High level rate-limit check is applied against the authenticated session. If its over the limit request is rejected at this point.

Step 6 and 7: With the help of a service discovery component, the AP gataway locates the appropriate backend service to handle the request by path matching.

Step 8: Api Gataway transforms the request into the appropriate protocol and send transformed request to the backend service.


When response comes back from the backend service, the API gataway transforms the response back to the public-facing protocol and returns the response to the client.

A proper API gataway also provides other critical services. For example: an API gataway should track errors and provides circuit breaking functionality to protect the services from overloading.

An API gataway should also provide logging, monitoring and analitics services for operational observability.

An API gataway is a critical piece of the infrastructure. It should be deployed to multiple regions to improve availability.

For many cloud provider offerings, the API gataway is deployed across the world close to the clients.

** What is Sharding
Instead of keeping everything in one place, we split the database into smaller, more managable pieces and distribute them across multiple servers. this is called sharding. 

We devide the database into smaller parts called shards. Each shard contains subset of total data. Data is distributed based on sharding key, for example user id. By distributing data this way, we reduce database load since each shard hadles only a portion of query and it speed ups read and write performance. Since querries are distributed across multiple shards, instead of hitting multiple database. Sharding is also referred as =Horizontal Partitioning= since it splits data by rows. But what if the issue is not rows but columns? In such cases, we use =Vertical Partitioning= where we split database by columns. Imagine we have a user table that stores profile details, login history and billing information. As this table grows querries becomes slower because table must scan many columns even when request only needs a few specific fields. To optimize this we use =Vertical Partitioning= where we split user table into smaller, more focused tables based on usage patterns. This improves query performance since each request only scans relevant columns instead of the entire table. It also reduces unnecessary disk i/o making data retrival quicker. 

No matter how much we optimize the database, retrieving data from disk is always slower than retrieving from memory. What if we store frequently accessed data in memory? This is called caching.

*** Caching

Caching is used to optimize the performance of a system by storing frequently accessed data in memory instead of repediately fetching it from database. One of the most common caching strategies is =Cache-aside pattern=. This is how it works: When a user request a data the application first checks the cache, if the data is in the cache, it's returned instantly avoiding the database call. If the data is not in the cache the application retrieves it from the database. It stores it in the cache for feature requests and returns it to the user. Next time, the same data is requested it serves directly from cache making the request much faster. To prevent outdated data from being served, we use time-to-live value or ttl.

*** Denormalization
Most relational database use normalization to store data efficiently by breaking it into seperate tables. While this reduces redundancy it also introduces joins. When retrieving data from multiple tables the data must combine them using join operations which can slow down queries as the data set grows.

Denormalization reduces number of joins by combining related data into a single table even if it means some data gets duplicated. For example instead of keeping users and orders in a seperate table, we create user orders table that stores user details along with the latest orders. Now when retrieving a user's order history, we don't need a join operation. The data is already stored together leading to faster querries and better read performance. Denormalization is often used in read-only applications where speed is more critical but the down side is it leads to increase the storage and more complex update requests.
